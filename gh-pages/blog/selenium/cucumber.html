Behaviour Driven Development Basics

Behaviour Driven Development is a software development approach where focus is on behaviour instead of the functionality.
  • For example: Customer wants to have login functionality on the application but the client doesn't know programming langauage so tells us the behaviour, how it should work.
  • So client says 
    			I am on the login page
			I enter user id and password and click on login button
			Then it should take me to home page
  • Taking this as requirement, developer writes code for each behaviour and tester writes functional code for each behaviour to test the same.
  • The above example consists of 3 must have steps Pre condition (at login page) , Action (give credentials and click), Result (Route to home page)
  • BDD uses the language Gherkin to define the behaviours. It's a plain text English language.
  • Discipline is very much required when opting for BDD approach. Collaboration is the core philosophy of this approach.
  • Earlier there were no common medium between DEV, QA, Stake holders but in BDD the developers use the same requirement written in Gherkin, QA refers the same to develop automation tests. Stake holders understand it too.

    • TDD Vs BDD

    Test Driven Development (TDD) is focused on test cases.
    • So when the requirement comes, the tests are designed first without knowing whether it will work or not.
    • Then the tests are run.
    • If it fails then developer writes the code.
    • Again the test runs. If it passes then its done if it fails then goes back to the developer again.
    • The process continues until the test case passes.
    This is often misunderstood with unit testing however that's different. Unit testing is what developer tests a single component where as TDD refers to execution of an actual test flow. This is also used by Agile teams. Some of the popular TDD tools are Junit, TestNG.

    Example: Promo code is made mandatory in the payment section. So first test if you keep it blank and try to make payment. If the flow works then the test fails. Developer will make the change until you are forced to add promo code before checking out.

    BDD is considered as an extension of TDD.
    • The requirement is the behaviour of the functionality written in plain english language.
    • Then corresponding to the behaviours functional code is written and tested.
    • If the test passes then its done else it goes back to the developer who makes the code change and it's tested again.
    • The process continues until the test passes.
    Example: Same example, the behaviour of the functionality will be. I am at the payments page, I keep promo code field as blank and try to make payment. Then it shouldn't check me out. Then functional code is developed and if the last behaviour fails, then developer will be asked to make code changes and then it will be tested again until all the behaviours are validated.

    Gherkin Language

    Gherkin (pronounced as Jerkin) is the BDD language.
    • BDD uses this plain text English language which can be understood by all.
    • We can also say Gherkin is a Business readable, domain specific language that is created to describe behaviors.
    • Using Gherkin we write behavior tests first and then implement functional codes later.
    • Gherkin has its syntax to standardize how it should be written
    • The Gherkin documents are stored as text files with .feature extension. 
    Feature: ATM Cash Withdraw
As an existing customer, I want to withdraw cash from ATM
    Scenario: Withdraw cash less than account balance
    Given I am at the ATM transaction screen
    When I entered the pin and the amount to withdraw
    Then the same amount cash is dispensed

    BDD in Agile

    The Agile & BDD complement each other beautifully as the core philosophy is collaboration for both. In agile the requirement is in the form of user story and in BDD the feature files are used. There are so many similarities so BDD fits perfectly in Agile model.
    • User story is nothing but a scenario in BDD feature file.
    • Since Agile tracks the progress as per the user story, it becomes easy to track the testing based on feature file scenarios.
    • Some teams write Agile user stories in Gherkin language so it becomes easy to use the same in BDD (cucumber automation)
    • Usually Product Owner (Business Analysts) write the user stories/gherkin feature files and the scrum team contributes to refine.



    Cucumber Basics

    Cucumber is a Behaviour Driven Development Framework. As the name indicates, this framework focuses on the behaviour of the feature and based on that everything else is driven.

    • Cucumber is currently owned by Smart Bear company
    • It is an open source tool however Cucumber Studio offered by Smart Bear is a licensed tool
    • Similar to cucumber there are different BDD tools available in market like JBehave, J Dave, FitNese etc
    • Cucumber is meant for automated user acceptance testing (UAT) as the business users can understand what is covered in testing.
    • Though its debatable where it should be used. This can be used anywhere it fits in. Most of the teams use it for Regression testing also.
    • Cucumber fits perfectly with Agile model as user stories can be converted into scenarios of feature files.
    • Cucumber was originally written in Ruby language.
    • Now cucumber supports many different languages like java, ruby, scala, java script, groovy etc.

    Feature File

    • A Feature File is an entry point to the Cucumber tests.
    • The file is saved with .feature extension.
    • This is the file where we write Gherkin steps (plain english)
    • This is an essential part of cucumber framework. This is a live documentation as well as executable requirement.
    • Feature file contains only scenarios (defined by the keyword scenario or scenario outline)
    • A single feature file can contain any number of scenarios but should be relevant to the feature such as registration, login etc.
    • The keywords in feature file - Feature, Given , When , Then , As, But, * , Scenario, Scenario Outline
    • Feature file is written by Business Analysts (Product Owners in Agile teams)
    • Some people call it feature steps or gherkin steps.

    Step Definition File

  • Step Definition is where the automation code is written.
  • The steps in the gherkin file directly map to the step definitions.
  • Each step in the gherkin maps to a single step definition that in turn contains a block of code that gets executed when the scenario is ran.
  • Step definition file contains these three things : Selenium code + Java code + Cucumber annotations like Given, Then, When etc.
  • Please note sequence of execution depends on the sequence of steps in feature file but not the sequence in step definition file.
  • Step definition file is developped by the automation testers in a project
  • Some people call it step definitions or some call it as glue code.

    • Test Runner

  • The runner file interacts with the feature file & the step definition file and executes the code.
  • By default Cucumber uses Junit framework to run. We can use TestNG also.
  • This class will use the Junit annotation @RunWith(), which tells JUnit what is the test runner class

    • Example

  • Create a maven project
  • Under src\test\java -> Create a new package features -> create a file Login.feature
  • Under src\test\java -> Create a new package steps -> create a file LoginSteps
  • Under src\test\java -> create a new package runner -> create a file LogInRunner

    • Login.feature 
    Feature:Login to application
Scenario: Valid login to application
Given I am at the login page
And Enter the user name as Biswajit
And Enter the password as Sundara
When click on login button
Then Verify the login is success

    LoginSteps.java 
    package steps;
import io.cucumber.java.en.Given;
import io.cucumber.java.en.Then;
import io.cucumber.java.en.When;
public class LoginSteps {
		
	@Given("I am at the login page")
	public void i_am_at_the_login_page() {
	   System.out.println("I am at the home page");
	}

	@Given("Enter the user name as Biswajit")
	public void enter_the_user_name_as_Biswajit() {
	   
		System.out.println("The user name is Biswajit");
	}

	@Given("Enter the password as Sundara")
	public void enter_the_password_as_Sundara() {
		System.out.println("The Password is Sundara");
	}

	@When("click on login button")
	public void click_on_login_button() {
	    
		System.out.println("Login button is clicked");
	}

	@Then("Verify the login is success")
	public void verify_the_login_is_success() {
		System.out.println("Log in success");
	}


}

    Run & Observe

    • Run the LoginRunner.java file. Right click -> Run as Junit Test
    • You will see while running its mapping the feature file steps with the corresponding step definitions.
    • If there's any step that's available in feature file but not in step definition file, then it will fail
    • We can also run the feature file, right click on Login.feature file -> Run as "Cucumber Feature" That's why it's called executable requirement.
    • The starting point is the feature file, so it's the most important file.

    Gherkin Keywords

    We know now that feature file should be written in Gherkin language and BDD/Agile operates on the philosohy of collaboration. So there should be a specific format or structure else everyone will have their own way of writing and which will lead to a messy document. That's where the keywords come in. The keywords are used to give structure and meaning to the specifications.

    1. Feature

    • Every Gherkin file starts with the keyword 'Feature'
    • Feature defines the functionality that we are going to test
    • E.g if we test payment gateway then Feature: Payment Gate Way
    • E.g if we test login functionality then Feature: Login
    • The idea of having a feature file is to put down a summary of what we will be testing.
    • Whatever comes after the Feature: keyword, till the next Keyword is encountered is considered as feature description. It can be a single line or multiple lines. 
      	Feature: LogIn Action Test
	OR
	Feature: LogIn Action Test
	Description: This feature will test a LogIn and LogOut functionality
	OR
	Feature: LogIn Action Test
	This feature will test a LogIn and LogOut functionality
	Please make sure to test everything.

    2. Scenario

    • Each test is called a scenario. It starts with the keyword 'Scenario'
    • In Agile terms we can say each user story is one Scenario.
    • A feature file can contain one or more scenarios.
    • Each scenario is broken into three logical parts - Pre condition (Given), Test (When), Verification (Then) 
      	Scenario: Search a product and add the first result/product to the User basket
	Or
	Scenario: Successful LogIn with Valid Credentials

    3. Given

    • Given defines a precondition to the test.
    • For Example: In shopping website, assume that the LogIn page link is only present on the Home Page. Then the precondition for clicking the LogIn link is that the user is at the Home Page. 
           Scenario: Successful LogIn with Valid Credentials
     Given User is on Home Page

    4. When

    • When keyword defines the user action.
    • In the below example the user 'clicks on the link' and that's an action. 
           Scenario: Successful LogIn with Valid Credentials
     Given User is on Home Page
     When User clicks on the log in link

    5. Then

    • Then keyword defines the Outcome of previous steps.
    • We can understand it better by looking at the test below and adding a Then step there. 
           Scenario: Successful Login with Valid Credentials
     Given User is on Home Page
     When User Clicks on the log in link
     And User enters UserName and Password
     Then Message displayed LogIn Successfully

    6. And

    And keyword is used to add conditions to your steps. Let's look at it by modifying our example a little 
         Scenario: Successful Login with Valid Credentials
     Given User is on Home Page
     When User Navigate to LogIn Page
     And User enters UserName and Password
     Then Message displayed Login Successfully
    
     OR
     
     Scenario: Successful Login with Valid Credentials
     Given User is on Home Page
     And LogIn Link displayed
     When User Navigate to LogIn Page
     And User enters UserName and Password
     Then Message displayed Login Successfully
     And LogOut Link displayed

    7. But

    But keyword is used to add negative type comments. It is not a hard & fast rule to use 'but' only for negative conditions.
    However it makes sense to use 'But' when you will try to add a condition which is opposite to the premise your test is trying to set. 
         Scenario: Unsuccessful Login with InValid Credentials
     Given User is on Home Page
     When User Navigate to LogIn Page
     And User enters UserName and Password
     But The user credentials are wrong
     Then Message displayed Wrong UserName & Password

    7. * Keyword

    This keyword is very special. This keyword defies the whole purpose of having Given, When, Then and all the other keywords. Basically Cucumber doesn't care about what Keyword you use to define test steps, all it cares about what code it needs to execute for each step. 
         Scenario: Successful Login with Valid Credentials
     Given User is on Home Page
     When User Navigate to LogIn Page
     And User enters UserName and Password
     Then Message displayed Login Successfully

Can be written as

     Scenario: Successful Login with Valid Credentials
     * User is on Home Page
     * User Navigate to LogIn Page
     * User enters UserName and Password
     * Message displayed Login Successfully




    Background

    Background in Cucumber is used to define a step or series of steps which are common to all the scenarios/tests in the feature file.
    • A Background is like a scenario containing one or multiple steps however it runs before each and every scenario in a feature file where it's defined.
    • Example: Let's say we are testing an e-commerce site for purchasing product or compare product price. Now for both of these scenarios we need to login to the application first: 
              Navigate to Login Page
	Enter UserName and Password
	Land on home page
      After these steps only we will be able to add a product to your cart/compare price/ able to perform the payment. So we won't repeat these steps in each scenario instead we can move it under Background so it gets executed before each scenario.
    • Background runs before each scenario, but after any Before hooks
    • Some people ask me if we have before hooks then what is the need of Background. It's quite simple, Before hook is not visible to customers or business users however Background is visible to them.
    • Business users read the background first and then they read the scenario to understand the complete flow.
    • In the feature file, Background should be positioned before the first Scenario.
    • The background is limited to one particular feature file. If we have multiple feature files then we will have to write background in all the feature files to make it work.
    • Background section should be short. Ideally not more than 4 lines as users need to remember this before reading the scenarios.
    • Don't write lengthy texts. It should be readable without scrolling right.
    Sample Feature file 
    
Feature: Online Shopping Feature
Description: The purpose of this feature is to make sure user is able to shop online

Background: User is Logged In
	Given I navigate to the login page
	When I enter username and password
	Then I should be on home page

Scenario: Search a product and add the first product to the user basket
	Given User search for iPhone 11 
	When Add the first iPhone that appears in the search result to the basket
	Then User basket should display with added item

Scenario: Search for products and compare the price
	Given User search for iPhone products 
	When compare all the product prices
        And Add the product with lesser price to the basket  
	Then User basket should display with lowest priced product
    In the above example, we have two different scenarios where user is adding a product from search and a lowest priced product. But the common step is to log In to website for both the scenarios. This is why we create another Scenario for Log In but named it as Background rather than a Scenario. So that it executes for both the Scenarios.



    Cucumber Options

    The runner file has annotation @CucumberOptions We can say this is the settings or runtime configurations for our test. We can specify the feature file path, step definition path, reporting path etc here. We can control the test execution by different parameters from here. This can be used only in the runner file. CucumberOptions is an interface under the cucumber.junit package.

    Dry Run

    • This option can either set as true or false (default value is false).
    • If it is set as true, it means that Cucumber will only check that every Step mentioned in the Feature File have corresponding code written in Step Definition file or not.
    • So in case any of the function is missed in the Step Definition for any Step in Feature File, it will give us the message.
    • So If you writing scenarios first and then implementing step definitions then add dryRun = true.
    • boolean dryRun() default false; is the method signature. 
      @CucumberOptions(features="src/test/java/features/login.feature", glue="steps" , dryRun= true)
    • Always use dryRun=true to generate the step definitions and later customize it.

    Strict

    • If strict option is set to false then at execution time if cucumber encounters any undefined/pending steps then cucumber does not fail the execution
    • Undefined steps are skipped and BUILD is SUCCESSFUL.
    • boolean strict() default false; is the method signature. 
      @CucumberOptions(features="src/test/java/features/login.feature", glue="steps" , strict= false)

    Feature

    • Features Options helps Cucumber to locate the Feature file in the project folder structure.
    • All we need to do is to specify the folder path and Cucumber will automatically find all the '.features' extension files in the folder.
    • It can be defined like: 
      @CucumberOptions(features="features", glue="steps" , strict= false)
      Here 'features' is the package name under which there are n number of feature files.
    • One very famous uestion is if I have 100 feature files, how can I run in cucumber? The answer is in the runner file, @CucumberOptions(features= specify the package name)
    • If we want to run only one feature file then we need to extend the path further so package name/file name features="src/test/java/features/login.feature"
    • String[] features() default {}; is the method signature

    Glue

    • This helps Cucumber to locate the Step Definition file.
    • Whenever Cucumber encounters a Step, it looks for a Step Definition inside all the files present in the folder mentioned in Glue Option.
    • In the base coding this is the signature of this = String[] glue()
      • Here 'steps' is the package under which we have the step definition files.
    • @CucumberOptions(features="src/test/java/features/login.feature",glue="steps" , strict= false)
      Here 'features' is the package name under which there are n number of feature files.

    Tags

    • Cucumber provides a simple method to organize features and scenarios by user determined classifications using tags.
    • Tag starts with “@”. After “@” you can have any relevant text to define your tag
    • In the feature file, you need to define the tags e.g = @smoketest, @unittest etc.
    • And then in the runner file you can mention the tag names to run only corresponding features. 
      @CucumberOptions(features="src/test/java/features/login.feature", glue="steps" , tags= {"@smoke"})
    • The base signature of the method = String[] tags()

    Plugin

    • Plugin Option is used to specify different formatting options for the output reports.
    • Various options that can be used as for-matters are:
      • A- Pretty: Prints the Gherkin source with additional colors and stack traces for errors.
      • B- HTML: This will generate a HTML report at the location mentioned in the for-matter itself.
      @CucumberOptions(features="src/test/java/features/login.feature", glue="steps" , plugin = {"pretty", "html:target/cucumber"})
    • The base signature = String[] plugin()
    • Format option is deprecated. Use Plugin in place of that.

    Monochrome

    • This option can either set as true or false (default value is false).
    • If it is set as true, it means that the console output for the Cucumber test is much more readable.
    • And if it is set as false, then the console output is not as readable as it should be.
    • For practice just add the code ‘monochrome = true‘ in TestRunner class. 
      @CucumberOptions(features="src/test/java/features/login.feature", glue="steps" , monochrome=true)
    • The base signature = boolean monochrome()




    Cucumber Tags

    We always need to categorize test cases for better maintainability. In TestNG this is called groups and in cucumber the same is called "tags"
    • Let's say you have got many different feature files which cover all the different functionality of the application. Now there can be certain situation in the project where you like to execute just a SmokeTests or End2EndTests or may be RegressionTests. One approach is that you create separate feature file which would make the project filthy.
    • Working with GitHub repo, we often need to enter the user name & password. If we use the SSH approach then we don't have to give username/password everytime.
    • For this, Cucumber has already provided a way to organize your scenario execution by using tags in feature file. We can define each scenario with a useful tag. Later, in the runner file, we can decide which specific tag (and so as the scenario(s)) we want Cucumber to execute. Tag starts with "@". After "@" you can have any relevant text to define your tag like @SmokeTests just above the scenarios you like to mark. Then to target these tagged scenarios just specify the tags names in the CucumberOptions as tags = {"@SmokeTests"}.
    • Tagging not just specifically works with Scenarios, it also works with Features. Means you can also tag your features files. Any tag that exists on a Feature will be inherited by Scenario, Scenario Outline or Examples.

    Different ways to use Tags

    • Cucumber can exclude scenarios with a particular tag by inserting the tilde character before the tag. 
      @CucumberOptions(features="src/test/java/features/login.feature",glue="steps" , tags= {"~@smoke"})
    • Tags can be separated by comma or logical OR 
      @CucumberOptions(features="src/test/java/features/login.feature", glue="steps" , tags= {"@smoke, @sanity"})
@CucumberOptions(features="src/test/java/features/login.feature", glue="steps" , tags= {"@smoke OR @sanity"})
    • Multiple tag arguments can be created with AND 
      @CucumberOptions(features="src/test/java/features/login.feature", glue="steps" , tags= {"@smoke AND @sanity"})
    • AND or NOT 
      @CucumberOptions(features="src/test/java/features/login.feature", glue="steps" , tags= {"@smoke,@sanity", "~@regression"})

    Cucumber Hooks

    Hooks are blocks of code that can run before or after every scenario. This is typically used for setup and teardown of the environment before and after each scenario.
    • We can declare hooks in any class in any package.
    • Cucumber hooks help us to better manage the test execution and also to avoid code duplication.
    • Often people get confused between Cucumber hooks & Background. So Hooks are unseen steps primarily focused on managing the resources however background is meant for taking us to a particular state from where a scenario can execute.
    • For example, logging into the application or logging out from the application should never be kept in hooks as these are steps business users to see. This should be kept under background.
    • However starting the web browser, initiating report logging or closing the DB connection should be under hooks as these should be kept under the covers and business users need not to see it.
    • Cucumber hooks supports only two annotation @Before, @After (don't get confused with TestNG which supports a whole bunch of annotations)
    • Hooks affect every scenario. it's good practice to keep hooks in a separate class.
    • Even though we can have Hooks in the glue code, it's not the best practice.
    • After hook executes always even in case the test fails. That's the reason why it's used in resource cleanup. Let's say we have opened the DB connection and started testing but the test failed. We will have memory leak and performance issue. Using the After hook, it ensures irrespective of the test pass/fail it will cleanup the resource by closing the DB connection in this example.
    • We can run the @before, @after methods in sequence in hooks file using order. @before(order=0), @before(order=1)

    Why do we need Cucumber Hooks

    Hooks can be used to perform background tasks that are not part of business functionality.
    Such prerequisite tasks could be
    • Starting up a browser
    • Setting or clearing cookies
    • Connecting to a database
    • Starting reports
    • Starting the error logger
    Same way some after test tasks could be
    • Closing browser
    • Closing the DB connection
    • Closing reports
    • Stopping the error logger

    Scenario Hooks

    Scenario hooks run for every scenario.
    @Before
    Methods annotated with @Before will execute before every scenario or we can say Before hooks run before the first step of each scenario. The below example starts browser before every scenario. 
    @Before
public void testInitialization() {
    startBrowser();
}

If we have multiple @Before annoted methods then we can specify the sequence how it should run.

@Before(order=2)
public void beforeScenario() {
    takeScreenshot();
}
    @After
    Methods annotated with @Before will execute before every scenario or we can say After hooks run after the last step of each scenario, even when the step result is failed, undefined, pending, or skipped. 
    @After
public void afterScenario() {
    closeBrowser();
}

    Step Hooks

    Step hooks are invoked before and after a step. If a BeforeStep hook is executed the AfterStep hooks will also be executed regardless of the result of the step. If a step did not pass, the following step and its hooks will be skipped.

    @BeforeStep
    Methods annotated with @BeforeStep execute before every step. Let's make an entry to the log file before each step. 
    public void beforeStep() {
    logStep();
}
    @AfterStep
    Methods annotated with @AfterStep execute after every step. 
    @AfterStep
public void afterStep() {
    takeScreenshot();
}

    Scenario parameter in Hooks

    The scenario parameter hold the information about the current scenario.
    • The getName method returns the scenario name.
    • The getId method returns the line number from the feature file when it is used in scenario. If we use the same in scenario outline, it will return the line number of the data.
    • The getStatus method returns the status of the scenario - PASSED/FAILED
    • If we invoke getStatus method in before method then it will return, status=UNDEFINED
    • If we invoke isfailed() in before method then it will return 'false'
    • Remember, the getStatus(), isfailed() are after method functions. 
    @After
public void afterScenario(Scenario scenario) { 
    // some code
}

    Example: Hooks.java

    I have placed the Hooks.java under the steps package. 
    package steps;
import cucumber.api.Scenario;
import cucumber.api.java.After;
import cucumber.api.java.Before;
public class Hooks {
	
	@Before
	public void beforeScenario(Scenario sc)
	{
		System.out.println(sc.getName());
		System.out.println(sc.getId());
		System.out.println("Before method is called in hooks");
	}

	@After
	public void AfterScenario(Scenario sc)
	{
		System.out.println(sc.getName());
		System.out.println(sc.getId());
		System.out.println(sc.getStatus());
		System.out.println(sc.isFailed());		
             System.out.println("After method is called in hooks");
	}
}

    Data Table

    Using step arguments we can pass data from feature file to step definition however if we have a huge set of data to supply then it won't be effective. For example let's say to fill a form we need data for 20-30 fields. If we write 20-30 lines in feature file and corresponding step definition then it will increase the code redundancy & maintenance will be difficult too. Cucumber data table allows us to pass data in the tabular format from feature file to the step definition. This works nicely and most users prefers this approach.
    • The data table is applied to one step only below which it is written.
    • The data table may or maynot contain the headers. Think like it's an html table only.
    • The table columns are separated by pipe symbol ( | )
    • The table may contain one or multiple rows/columns.
    • I don't think there's any maximum count of rows/columns that we can have in data table. However cucumber feature file should be mantained in a single page view. So scrolling much horizontally or vertically should be avoided. Instead long steps should be broken into multiple steps.
    • The data table can be one or multiple rows/columns. With or Without headers
    • The reason why we need headers in data table columns is it gives meaning to the column. Without headers we don't know which data the column holds.

    Cucumber & TestNG

    Cucumber can be integrated with TestNG and most of the hybrid automation frameworks use this approach. By default Cucumber supports Junit however integrating with TestNG we can use more features provided by TestNG. So We need to make three major updates-
    • In Maven (POM.xml) add cucumber-testNG, TestNG dependency
    • In CucumberRunner / TestRunner file extends AbstractTestNGCucumberTests
    • In TestNG.xml add path and name of CucumberRunner / TestRunner file in class tag.

    Step1: Update pom.xml file

    In pom.xml add the dependencies for cucumber-testng & the testng. This will download the supporting jar files. 
    
<!-- https://mvnrepository.com/artifact/io.cucumber/cucumber-testng -->
  <dependency>
	<groupId>io.cucumber</groupId>
	<artifactId>cucumber-testng</artifactId>
	<version>5.3.0</version>
  </dependency>

<!-- https://mvnrepository.com/artifact/org.testng/testng -->
  <dependency>
	<groupId>org.testng</groupId>
	<artifactId>testng</artifactId>
	<version>6.14.3</version>
  </dependency>

    Step2: Update TestRunner file

    The TestRunner class file should extend the class AbstractTestNGCucumberTests. This will allow cucumber tests to be run on TestNG instead of JUnit. 
    
import org.testng.annotations.AfterClass;

	public void beforeMethod()

	}
	
	@AfterSuite
	public void afterSuite()
	{
		System.out.println("after suite step executed");
	}
	
	@AfterClass
	public void afterClass()
	{
		System.out.println("after class step executed");
	}
	
	@AfterTest

    Step3: Configure TestNG.xml file

    <?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE suite SYSTEM "http://testng.org/testng-1.0.dtd">
<suite name="Suite">
  <test thread-count="5" name="Test">

    Observations

    Run the testng.xml file and see how it works.
    • Before Suite is executed first, before any runner file.
    • Before test is executed then, before the runner file.
    • Before class is executed before executing the runner file.
    • Before method is executed before the scenario in feature file.
    • After method is executed after the scenario in feature file.
    • For scenario outline, it's basically number of scenarios so for each scenario it executes before method & after method.
    • Once the runner file is completed. executes after test
    • Then finally after suite.
    • Test reports are available at test-output folder.

    package base;

import java.io.File;

import com.aventstack.extentreports.ExtentReports;
import com.aventstack.extentreports.ExtentTest;
import com.aventstack.extentreports.GherkinKeyword;
import com.aventstack.extentreports.MediaEntityBuilder;
import com.aventstack.extentreports.MediaEntityModelProvider;
import com.aventstack.extentreports.reporter.ExtentHtmlReporter;

public class CustomExtentReporter {

	public ExtentHtmlReporter htmlReport;
	public ExtentReports extentReport;
	public ExtentTest extentTestStep,extentTestCase, extentTest;


	public void startReport(String reportLocation)
	{
		htmlReport= new ExtentHtmlReporter(new File(reportLocation));
		extentReport = new ExtentReports();
		extentReport.attachReporter(htmlReport);
	}

	public void startTestCase(String testcaseName)
	{
		extentTestCase = extentReport.createTest(testcaseName);
	}
	
	public void startTestNode(String testNode)
	{
		extentTestStep=extentTestCase.createNode(testNode);
	}
	
	public void reportTestStep(GherkinKeyword ghKeyword, String name)
	{
		extentTest = extentTestStep.createNode(ghKeyword, name);
	}
	
	public void reportTest(String desc, String status, String imgPath)
	{
		try
		{
			MediaEntityModelProvider img = null;
			img=MediaEntityBuilder.createScreenCaptureFromPath(imgPath).build();
			
			if(status.equalsIgnoreCase("PASS")) 
			{
				extentTest.pass(desc);
			}
			else if (status.equalsIgnoreCase("FAIL")) 
			{
				extentTest.fail(desc, img);
			}	
		}
		catch(Exception e)
		{
			System.err.println(e.getMessage());
		}
	}
	
	public void saveReport()
	{
		if(extentReport!=null)
		{
			extentReport.flush();
		}
	}
}

    Hook.java

    package steps;

import java.io.IOException;

import org.openqa.selenium.WebDriver;

import base.TestBase;
import io.cucumber.java.After;
import io.cucumber.java.Before;
import io.cucumber.java.Scenario;

public class Hook {

	private WebDriver driver;

	@Before
	public void InitializeTest(Scenario scenario) {
		TestBase.setWebDriver();
		driver= TestBase.getWebDriver();
				
		TestBase.getReport().startTestCase(scenario.getName());
		TestBase.getReport().startTestNode(scenario.getName());
		
	}
	
	
	@After
	public void TearDownTest(Scenario scenario) throws IOException {
		String screenShotFileName= "C:\\snaps" + scenario.getName().replaceAll(" ", "")+".png";
		if(scenario.isFailed())
		{
			TestBase.takeScreenShot(screenShotFileName);
			TestBase.getReport().reportTest(scenario.getName(), "FAIL", screenShotFileName);
		}
		else
			
			TestBase.getReport().reportTest(scenario.getName(), "PASS", screenShotFileName);
		
		
		if(driver!=null)
		driver.quit();
	}
	

}

    Dependency Injection

    When it's required to share state between cucumber steps we need to use Dependency Injection. As the name implies it injects the current state to the step definitions. 

    Feature: SearchFeature
  
This feature is validate user is able to search product

  Scenario: Search Product
    Given I am at the login page
    And I enter the login credentials
      | username       | password   |
      | abc@gmail.com  | 123        |
    And I click login button
    And I am at the home page
    And I search for dress
    And I click on search button
    Then I should see the product names

    If we look at the above scenario, it deals with multiple step definition files such as loginSteps, homepageSteps, searchSteps. Now the question is when it executes the methods from all these three files how is it going to operate on the same web driver instance. The answer is dependency injection. We need to create WebDriver at a common place and then inject it in all these three files so during run time it should refer to the same webdriver instance. Another example is if we want to share data between steps. Let's say we get the confirmation number in one step and then want to verify the number on the next page using another step.

    Pico Container

    Using pico container we can achieve dependency injection. Add the below dependency to pom.xml 
    <dependency>
    <groupId>org.picocontainer</groupId>
    <artifactId>picocontainer</artifactId>
    <version>2.15</version>
</dependency>

    1. Cucumber Best Practices


    Prevent leaking state between scenarios

    • Avoid using global or static variables.
    • Make sure you clean your database in a Before hook
    • If you share a browser between scenarios, delete cookies in a Before hook

    Independent scenarios

    The scenarios should be independent to each other. For example: Some beginners write login as one scenario and then purchase as another scenario and end up with errors. The purchase scenario itself should have the login step before purchasing the product or we can keep login under background.

    Data table size

    The data table size should be kept to minimum. There's no finite number as how many columns we can keep however the best practice is to keep in such a way that the reader doesn't have to scroll too much. Usually 8-10 columns should be enough. If the data table grows big then the good approach is to use external files such as JSON or Excel files and mention the file names in the data table.
